Runs, history, and troubleshooting

Updated July 10, 2026

Every execution of a flow is a run. This article covers how runs start, the waiting state on episode uploads, the history log, and the most common failures.

#How it works

A run is a single pass through a flow's saved graph, tracked as its own record with a status and a step-by-step log. The engine starts at the trigger, executes each connected node once, and follows the outgoing edges whose branch filters pass, so a run with a router can fan out into more than one path in the same execution. The log it produces is what powers both the History panel below and the live progress screen you see the first time you add an episode.

#How runs start

  • Automatically. An Episode Upload flow with its Active switch on queues a run whenever a new episode is created in the workspace. Every new workspace starts with one such flow already active: First Content Pack. Its run is what you watch build on the onboarding screen the first time you add an episode.
  • Manually from the Flows page. Click Run on the flow card, pick an episode (uses its transcript) or a saved script (uses the script text), then click Run flow.
  • Manually from the editor. Enter an episode's ID in Episode ID for Run and click Run. Without an ID the run starts with no source, and steps that need one are skipped.

A flow must be saved before it can run. In a brand-new flow, Run stays disabled until the first Save.

#The waiting state

Episode-upload runs do not start the moment the episode is created. The transcript is still being generated at that point, and the AI nodes need it. So the run is held in a Waiting state, shown as an amber badge on the flow card.

  • Once the episode's transcript completes, Vidflow waits a few more minutes for metadata to finish populating, then starts the run.
  • If the episode is not ready within about 30 minutes of upload, or transcription fails, the run is marked failed and you get a notification. Fix the transcript first (see Transcripts), then run the flow manually.

Manual runs skip the wait entirely and start right away, because you picked a source that already exists.

#Run statuses

A run moves through these statuses, visible on the flow card and in the history:

  • waiting: held until the episode's transcript and metadata are ready (episode-upload runs only).
  • pending: accepted and about to start.
  • running: in progress. The flow card shows a violet Running badge.
  • completed: finished.
  • failed: stopped with an error. The history shows the error text.

#Reading the history

Open the flow card menu and choose History. The Execution History panel lists recent runs, newest first, and refreshes every few seconds while open. Each entry shows the status, the trigger type, when it ran, and the duration.

Click a run to expand its step log. Each step shows the node, its outcome, and a message. Steps have three outcomes:

  • completed: the node did its work, often with a count (posts written, items saved).
  • skipped: the node had nothing to act on and the run moved past it. Examples: no episode in context, nothing upstream to save, or a Condition that did not pass.
  • failed: the node hit an error. Failed generation steps include the reason.

#Troubleshooting

The run failed with "Episode wasn't ready within 30 minutes of upload" The transcript did not finish inside the wait window. Open the episode, check its transcription status, and retry the transcript if needed. Once it is ready, run the flow manually against that episode.

The run failed with "Episode transcription failed" Retry transcription from the episode, then run the flow manually.

Steps were skipped with "No episode in context" The run started without a source, usually from the editor with no Episode ID for Run. Run the flow from the Flows page instead and pick an episode or script.

The run completed but nothing is in the library. The flow has no Save node. Generated drafts that never reach Save to Library, Save to Queue, or a Post node are discarded when the run ends. Add one and run again.

Starting a run fails with "flow runs in progress" Vidflow caps concurrent runs at three per workspace, because each run can fan out into many AI and clip jobs. Wait for the running flows to finish, then start the next one.

A generation step failed with a monthly limit message. The workspace has used up its plan's monthly AI or clip allowance. See Usage limits and monthly allowances for how allowances work and what to do at the cap.

The flow never fires on new episodes. Confirm the trigger is Episode Upload and the Active switch on the flow card is on. Google Drive flows do not fire on their own during the beta.

A run looks stuck on Running. Clip generation and image steps can take a while. Check the step log in History to see which node is active. If a run has clearly stalled for a long time, run the flow again or contact support at support@vidflow.io with the workspace name and the run's timestamp.

#Get better results

  • Check History after every new flow's first couple of runs, not just when the library looks wrong. A run can report "completed" while a branch quietly skipped, and the step log is the only place that shows why.
  • When testing changes to a flow, run it manually against an episode you already know has a finished transcript. This skips the waiting state entirely and gets you a result in the time it takes the AI and clip nodes to run.
  • If you regularly hit the three-concurrent-runs cap, spread manual runs out instead of queuing several back to back, or wait for one to clear History before starting the next.
  • Treat a "skipped" step as information, not a failure. It usually means a condition correctly filtered something out or a node genuinely had nothing upstream to act on.
